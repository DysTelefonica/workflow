    • 1. ¿Qué es Dysflow?

    Dysflow es un framework de desarrollo basado en especificaciones (SDD — Spec-Driven Development) diseñado para proyectos Microsoft Access con VBA.

    🎯 Objetivos principales

  • Trazabilidad total: Cada cambio parte de una especificación documentada
  • Cero regresiones: Análisis de impacto antes de modificar código
  • Entrega manual: El código se copia manualmente al editor VBA
  • Calidad arquitectónica: Separación MVVM, transacciones explícitas

    • 🏗️ Arquitectura del sistema

    El sistema funciona con un agente de IA (Trae, OpenCode, Cursor) que:

    • Lee y escribe código VBA
    • Gestiona especificaciones y PRDs
    • Mantiene la memoria entre sesiones (Engram)
    • Orquesta el flujo de desarrollo completo

    2. Instalación

    📋 Requisitos previos

    Antes de instalar Dysflow, asegúrate de tener:

    Windows 10/11 Node.js 18+ Git 2.30+ Access 2016+ PowerShell 7+

    ⚠️ Importante

    Dysflow solo funciona en Windows porque necesita Microsoft Access y PowerShell.

    Verificar versiones

    PS> node --version
    PS> git --version
    PS> pwsh --version

    🚀 Instalación con npx

    Dysflow se ejecuta directamente desde GitHub usando npx. No necesitas clonar ni instalar nada localmente.

    1

    Verificar que tienes Node.js

    PS> node --version

    Debe mostrar versión 18 o superior.

    2

    ¡Listo! No hay más pasos

    Dysflow se ejecuta directamente con npx.

    💡 ¿Por qué npx?

    npx descarga y ejecuta el paquete sobre la marcha. Siempre tendrás la última versión sin necesidad de actualizar.

    ✅ Verificar instalación

    Ejecuta este comando para verificar que Dysflow funciona:

    PS> npx github:DysTelefonica/workflow --help

    Deberías ver:

Commands:
  spec <number>      Create spec branch
  plan <number>      Create plan branch
  release             Create release from develop
  hotfix <name>      Create hotfix branch

  next-release       Get next release tag
  init <type>        Initialize project (access)

    ✅ ¡Instalación completada! Continúa con la siguiente sección.

    4. El Flujo SDD — Spec-Driven Development

    El corazón de Dysflow. Todo desarrollo sigue este flujo con 4 fases y 2 STOPs.

    Historia de Usuario
    ↓
    Fase 1: Análisis → Spec
    ↓
    STOP 1: Aprobación
    ↓
    Fase 2: Implementación
    ↓
    STOP 2: Validar en Access
    ↓
    Fase 4: Cierre

    Fase 1 — Análisis y Spec

    Cuando el usuario describe lo que quiere (historia de usuario, bug, mejora):

  • Buscar en Engram (memoria persistente) contexto previo
  • Leer DISCOVERY_MAP para localizar módulos afectados
  • Leer PRD del módulo
  • Analizar código fuente si es necesario
  • Detectar si es una Epic (>3 módulos afectados)
  • Generar la Spec en docs/specs/active/

    • Fase 2 — Implementación

    Una vez aprobada la Spec:

  • Leer la Spec aprobada
  • Implementar cada intervención
  • Aplicar convenciones de código
  • Auto-verificar contra criterios de la Spec
  • Generar Informe de Cambios UI si hay cambios en formularios

    • STOP 1 — Aprobación de Spec

    ⏸️ La IA se detiene aquí

    El usuario revisa:

    • ¿El análisis de impacto es correcto?
    • ¿Las intervenciones cubren todo lo necesario?
    • ¿Los criterios de verificación son adecuados?

    Si pide cambios → modificar Spec y volver a presentar.

    Si aprueba → pasar a Fase 2.

    STOP 2 — Validación en Access

    ⏸️ La IA se detiene aquí

    El usuario:

    1. Copia los módulos a su proyecto VBA/Access
    2. Compila y prueba
    3. Responde con:
    • VALIDADO EN ACCESS: Spec-XXX → ir a Cierre
    • Descripción de un gap → ir a Fase 3 (Iteración)

    Fase 4 — Cierre

    Cuando el usuario confirma VALIDADO EN ACCESS: Spec-XXX:

  • Archivar Spec en completed/
  • Actualizar estado a ✅ VALIDADO EN ACCESS
  • Crear o actualizar PRD
  • Actualizar DEUDA_TECNICA.md
  • Revisar DISCOVERY_MAP
  • Registrar en Diario de Sesiones
  • Guardar en Engram
  • Actualizar CHANGELOG

    • 5. Casos de Uso

    Ejemplos prácticos de cómo usar Dysflow en situaciones reales.

    📝 Nueva Funcionalidad

    El usuario dice:

    Usuario
    "Yo como miembro de calidad quiero que cuando registre una inspección, el sistema me avise automáticamente si el expediente ya tiene una inspección cerrada en el último mes, para evitar duplicidades."

    Qué sucede:

    1

    Fase 1: Análisis

    La IA busca en memoria, lee DISCOVERY_MAP, PRD del módulo de inspecciones, analiza código.

    2

    Genera Spec

    Crea docs/specs/active/spec-042-evitar-duplicidades/Spec-042_EvitarDuplicidades.md

    3

    STOP 1

    Presenta la Spec al usuario para aprobación.

    4

    Implementa

    Implementa cada intervención, auto-verifica.

    5

    STOP 2

    Presenta módulos modificados. El usuario copia a Access y prueba.

    6

    Cierre

    Usuario dice: VALIDADO EN ACCESS: Spec-042

    La IA archiva, actualiza PRD, guarda en Engram, actualiza CHANGELOG.

    🐛 Bug Urgente en Producción

    El usuario dice:

    Usuario
    "Hay un bug crítico: cuando se elimina un expediente, los registros de inspecciones asociados quedan huérfanos. Hay que corregirlo ya."

    Qué sucede:

    1

    Crear rama hotfix

    PS> npx github:DysTelefonica/workflow hotfix fix-huerfanos-inspeccion

    Crea rama hotfix-fix-huerfanos-inspeccion desde main.

    2

    Spec con prefijo "hotfix"

    La IA genera una Spec con título "Hotfix: Corregir registros huérfanos".

    3

    Implementar y validar

    Mismo flujo que funcionalidad normal.

    4

    Cierre manual

    El usuario ejecuta los comandos de merge manualmente (el hotfix no pasa por develop).

    🏗️ Cambio de Arquitectura

    El usuario dice:

    Usuario
    "Queremos migrar el sistema de gestión de usuarios a un patrón repositorio limpio. Actualmente está todo en los formularios."

    Qué sucede:

    1

    RFC primero

    Se crea un RFC (Request for Comments) documentando el cambio de arquitectura. El usuario lo aprueba.

    2

    Detectar Epic

    La IA detecta que el cambio afecta >3 módulos. Ofrece crear un Plan de Actuación.

    3

    Plan de Actuación

    Plan-043: Migración RepositorioUsuario
    Contiene: Spec-044, Spec-045, Spec-046

    4

    Spec por Spec

    Spec-044: Extraer RepositorioUsuario
    Spec-045: Crear ServicioUsuario
    Spec-046: Refactorizar formularios

    6. Git Workflow

    Dysflow usa un Git Flow simplificado para gestionar ramas y releases.

    🌿 Ramas del proyecto

    main
    Código en producción
    develop
    Integración de features
    spec-NNN-*
    Implementación de specs
    hotfix-*
    Correcciones urgentes

    📋 Convención de nombres

    Tipo Formato Ejemplo
    Spec spec-{NNN}-{slug} spec-042-evitar-duplicidades
    Hotfix hotfix-{nombre} hotfix-fix-login
    Release YYYY-NNN 2026-003

    🔄 Flujo de desarrollo

    develop
    →
    spec-042
    →
    develop
    →
    main + tag

    📌 Comandos de ramas

    Acción Comando
    Crear rama para Spec dysflow spec 042
    Crear rama para Hotfix dysflow hotfix fix-login
    Crear release dysflow release

    7. Comandos CLI

    Todos los comandos disponibles en Dysflow:

    📌 Cómo ejecutar los comandos

    Usa npx github:DysTelefonica/workflow seguido del comando:

    PS> npx github:DysTelefonica/workflow init access
    Comando Descripción
    dysflow spec <number> Crear rama spec-{NNN}-{slug} desde develop
    dysflow plan <number> Crear rama plan-{NNN}-{slug} para Planes de Actuación
    dysflow release Fusionar develop → main, crear tag YYYY-NNN
    dysflow hotfix <name> Crear rama hotfix desde main
    dysflow changelog <from> Generar changelog desde tag
    dysflow next-release Mostrar el próximo número de release
    dysflow init access Inicializar proyecto Access
    dysflow update Actualizar skills y rules

    8. Skills del Sistema

    Las skills son capacidades que el agente de IA puede ejecutar:

    Skill Función
    🎯 sdd-protocol Orquestador del flujo SDD
    📝 spec-writer Generación de especificaciones técnicas
    📋 prd-writer Generación y mantenimiento de PRDs
    🐛 hotfix Gestión de bugs urgentes
    🏗️ rfc-writer Cambios de arquitectura
    🔄 access-vba-sync Sincronización Access ↔ código
    📊 plan-writer Planes de actuación para Epics
    📔 diario-sesion Registro de sesiones de desarrollo

    9. Memoria Persistente (Engram)

    El sistema usa Engram para recordar decisiones entre sesiones:

    Función Cuándo usarla
    mem_search Buscar contexto previo antes de actuar
    mem_save Guardar decisiones importantes (arquitectura, bugs)
    mem_session_start Iniciar sesión (al principio de cada sesión)
    mem_session_summary Cerrar sesión con resumen (al final, obligatorio)

    💡 Regla de oro

    Engram primero: Antes de leer cualquier archivo, buscar en Engram. El conocimiento ya aprendido no se re-aprende.

    11. Resolución de Problemas

    ❌ "Access is being used by another process"

    Solución: Cerrar Access completamente y volver a intentar.

    PS> node skills/access-vba-sync/cli.js generate-erd --backend "C:\Ruta\MiBackend.accdb"

    ❌ "No se encontró la rama develop"

    Solución: Crear develop desde main si no existe.

    PS> git checkout main
    PS> git checkout -b develop
    PS> git push -u origin develop

    ❌ La Spec no pasa el STOP 1

    Solución:

    • Revisar el análisis de impacto
    • Aclarar las ambigüedades en "Gaps y Decisiones"
    • Añadir más contexto en la historia de usuario
